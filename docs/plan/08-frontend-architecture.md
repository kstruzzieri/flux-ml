# Frontend Architecture

## Overview

Flux uses a lightweight, performance-focused frontend architecture optimized for a desktop application with a known, fixed UI. The architecture prioritizes simplicity, zero runtime overhead, and maintainability.

## CSS Strategy

### Why Vanilla CSS

We use vanilla CSS with CSS custom properties (design tokens) rather than a CSS framework. This decision reflects the project's constraints and goals:

| Factor | Framework (e.g., Tailwind) | Vanilla CSS |
|--------|---------------------------|-------------|
| Bundle size | 10-50 KB+ | 0 KB overhead |
| Runtime cost | Minimal to moderate | Zero |
| Build complexity | Additional tooling | Native browser support |
| Learning curve | Framework-specific | Universal CSS knowledge |
| Desktop app fit | Overkill for fixed UI | Purpose-built |

**Key considerations:**

1. **Performance target:** Near 0% idle CPU means no runtime CSS processing
2. **Known UI:** Unlike a website with many page variations, Flux has a fixed panel layout
3. **Binary size:** Every KB matters for a ~15MB target
4. **Simplicity:** Fewer dependencies = fewer failure points

### Design Tokens

Design decisions are captured as CSS custom properties in `styles/tokens.css`:

```css
:root {
  /* Colors */
  --color-bg-base: #06090c;
  --color-bg-surface: #0d1117;
  --color-bg-elevated: #161b22;

  /* Spacing */
  --spacing-xs: 4px;
  --spacing-sm: 8px;
  --spacing-md: 16px;

  /* Typography */
  --font-ui: 'Inter', sans-serif;
  --font-mono: 'JetBrains Mono', monospace;
}
```

Benefits:
- Single source of truth for design values
- Enables future theme switching (dark/light)
- Self-documenting code
- Easy global updates

### BEM Naming Convention

We use BEM (Block Element Modifier) for CSS class naming to provide structure without framework overhead:

```
.block              → Standalone component
.block__element     → Part of a block
.block--modifier    → Variation of block/element
```

Example:
```css
.sidebar { }                    /* Block */
.sidebar__title { }             /* Element */
.sidebar__item { }              /* Element */
.sidebar--collapsed { }         /* Modifier */
.sidebar__item--active { }      /* Element modifier */
```

Benefits:
- Flat specificity (no nested selectors)
- Self-documenting class names
- Predictable cascade behavior
- Easy to search/grep

## Component Structure

### Directory Layout

```
frontend/src/
├── components/
│   ├── layout/              # Shell and panel components
│   │   ├── AppShell.tsx
│   │   ├── Header.tsx
│   │   ├── LeftSidebar.tsx
│   │   ├── MainContent.tsx
│   │   ├── RightInspector.tsx
│   │   ├── BottomPanel.tsx
│   │   └── index.ts
│   ├── common/              # Shared UI components (future)
│   └── index.ts
├── styles/
│   ├── tokens.css           # Design tokens
│   ├── reset.css            # CSS reset/normalize
│   └── components/          # Component-specific styles
│       └── layout.css
├── hooks/                   # Custom React hooks
├── stores/                  # State management (future)
└── main.tsx
```

### Component Principles

1. **Single responsibility:** Each component does one thing well
2. **Semantic HTML:** Use proper ARIA roles and landmarks
3. **Props over state:** Prefer controlled components
4. **Co-located styles:** CSS files mirror component structure

### Accessibility

Layout components use semantic HTML and ARIA landmarks:

| Component | Role | Element |
|-----------|------|---------|
| Header | `banner` | `<header>` |
| LeftSidebar | `complementary` | `<aside aria-label="Sidebar">` |
| MainContent | `main` | `<main>` |
| RightInspector | `complementary` | `<aside aria-label="Inspector">` |
| BottomPanel | `contentinfo` | `<footer>` |

## State Management

For Phase 1 (Foundation), we use React's built-in state:
- `useState` for local component state
- `useContext` for shared state (theme, layout preferences)

Future phases may introduce a state library if complexity warrants it.

## Wails Integration

### Go-to-JS Bindings

Go methods exposed to the frontend are accessed via generated TypeScript bindings:

```typescript
// Generated by Wails in wailsjs/go/main/App.ts
import { GetAppInfo } from '../../wailsjs/go/main/App';

const info = await GetAppInfo();
// { name: "Flux", version: "0.1.0" }
```

### Binding Conventions

- Go methods are PascalCase
- Return structured data (maps/structs), not primitives
- Handle errors in Go, return clean data to frontend
- Use TypeScript types for returned data

## Performance Guidelines

1. **No polling:** Use event-driven updates from Go backend
2. **Lazy loading:** Load panel content on demand
3. **Memoization:** Use `useMemo`/`useCallback` for expensive operations
4. **CSS containment:** Use `contain` property for isolated panels
